# 企业微信真实收发流程

目标是不改变供应商的使用习惯：供应商仍然在企业微信会话里直接发图片和文字，系统负责自动收图、入库、生成报表，并把选款和寄样流程继续推进。

## 推荐架构

### 1. 主链路：企业微信官方能力

优先使用企业微信官方能力，而不是长期依赖桌面 UI。

- `会话内容存档`：适合归档员工和外部联系人的真实聊天记录，能覆盖供应商直接发来的图片。
- `客户联系接口`：适合同步外部联系人，并获得官方 `external_userid`。
- `接收消息与事件回调`：适合自建应用消息入口。
- `应用消息接口`：适合给内部成员发送报表和异常提醒。

落地流程：

```text
拉取或接收消息
-> 解析发送人 external_userid
-> 映射到供应商/选款人/运营角色
-> 下载图片媒体
-> 写入 inbox event
-> 入库、分类、生成报表
-> 继续推进状态机
```

### 2. 兜底链路：桌面端企业微信

桌面端适合测试期和部分对外发送场景：

- 给供应商发送每日问款。
- 给选款人发送报表。
- 给供应商发送寄样和商品信息请求。
- 给运营发送结构化商品信息表。
- 官方接口不可用时保留人工接管能力。

桌面端不能作为唯一监听手段，因为它依赖电脑不睡眠、企业微信不掉线、窗口可见、UI 不变化。

### 3. 桌面发送与官方收图的身份映射

如果发送走桌面端、收图走官方 SDK，同一个联系人会有两个标识：

```text
桌面端：会话名/搜索名
官方 SDK：external_userid
```

系统不能假设这两个标识天然一致。正确做法是：

1. 桌面问款消息带唯一批次码，例如 `今日批次：0602-SUPPLIER01`。
2. SDK 存档里找到同一条 outgoing 消息。
3. 从 outgoing 消息的收件人列表取到官方 `external_userid`。
4. 把这个 `external_userid` 绑定到本地联系人。
5. 没有 SDK outgoing 证明的联系人，不应自动进入后续业务流程。

这能避免多个供应商同一时间收到完全相同文案时发生错绑。

### 4. 长期方向：收发都走官方接口

最稳定的方案是发送和接收都使用官方接口：

```text
发送：external_userid
接收：external_userid
```

这样不再需要从桌面会话名翻译到官方联系人 ID。这个方向通常需要企业微信客户联系权限、可信 IP、域名、合规授权和可能的 ICP 备案周期。

## 状态流转

每个供应商每天有一条状态：

```text
pending_ask -> ask_sent -> waiting_images -> images_received -> report_ready
-> report_sent -> selection_received -> sample_requested
-> supplier_info_received -> ops_table_sent -> done
```

异常情况会进入人工确认或恢复前对账：

```text
unknown_sender
missing_external_userid
desktop_send_failed
receive_channel_unhealthy
selection_uncertain
supplier_info_incomplete
```

## 安全原则

- 不自动翻个人微信朋友圈。
- 只处理供应商主动发来的图片/文字，或正式沟通入口中的业务消息。
- 不把内部商品编号发给供应商。
- 截图圈选识别不确定时进入人工确认。
- 涉及改地址、付款、下单承诺、价格确认、换款、取消等高风险动作时进入人工确认。
- 真实联系人、密钥、消息、图片和日志不进入公开仓库。
